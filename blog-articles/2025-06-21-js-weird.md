# JavaScript oddities that make more sense than they seem

> "JavaScript is charmingly offbeat."  
> The Primeagen

Sometimes people talk about the "WATs" of JavaScript. You do not need to memorize all of them to build useful software, but knowing how some of these behaviors work can save you a lot of debugging time.

When a strange bug appears, it is easy to blame the language. In practice, the real issue is often that our mental model was incomplete. JavaScript certainly has historical quirks, but many of them become understandable once you know the rules behind them.

## Weird doesn't always mean random

If you think you never rely on type coercion, that is probably not true. Everyday JavaScript uses coercion all the time: template strings, string concatenation, console logging, form values, comparisons, and conditional checks.

That does not mean coercion is always good. It means that understanding it is useful.

The same applies to things like Automatic Semicolon Insertion (ASI). You may not care about it most of the time, until one line break after a `return` turns into a very confusing bug.

## Truthy and falsy values

Let us start with the basics. Some values are truthy, which means they become `true` in a boolean context:

```js
Boolean("false") // true
Boolean({}) // true
Boolean(() => null) // true
Boolean([]) // true
Boolean(" ") // true
Boolean(-1) // true
Boolean(new Boolean(false)) // true
```

And some values are falsy:

```js
Boolean("") // false
Boolean(0) // false
Boolean(null) // false
Boolean(undefined) // false
Boolean(NaN) // false
```

That alone explains a lot of conditionals that feel strange at first.

## The `+` operator is where many surprises begin

A few simple examples:

```js
101 - "1" // 100
101 + "1" // "1011"
101 + {} // "101[object Object]"
```

With `-`, JavaScript tries numeric conversion. With `+`, if one side becomes a string, concatenation happens.

Arrays follow the same pattern because they can be converted to strings:

```js
[1, 2, 3] + [4, 5, 6] // "1,2,34,5,6"
```

The arrays are converted to `"1,2,3"` and `"4,5,6"`, and then concatenated.

Truthy and falsy values also behave numerically in arithmetic:

```js
true + true + true // 3
2 * true // 2
(true + true) * (true + true) - true // 3
```

That is why this is true:

```js
!"true" == !"false" // true
!"true" === !"false" // true
```

Both strings are truthy, both become `true`, and negating them gives `false`.

## Arrays and objects can look especially strange

```js
[] + 1 // "1"
[] + "1" // "1"
[] - 1 // -1
[] - "1" // -1
```

An empty array becomes an empty string in string contexts, and `0` in numeric contexts.

Objects are even trickier because `{}` can be parsed differently depending on context:

```js
({} + 1) // "[object Object]1"
({} - 1) // NaN
```

If you try these directly in the console without parentheses, parsing rules may change the result. That is one reason many "weird JS" examples are confusing: some are really about syntax context, not just values.

Now compare these:

```js
[] == false // true
Boolean([]) // true
[] == {} // false
[] + [] // ""
[] + {} // "[object Object]"
({} + []) // "[object Object]"
[] == ![] // true
```

`[] == false` is a classic one. The array is truthy in a boolean context, but with loose equality JavaScript applies coercion rules that eventually compare `0 == 0`.

## `==` and `===` are not the same problem

JavaScript has two equality operators:

```js
0 == false // true
0 === false // false
null == undefined // true
null === undefined // false
```

If the types already match, `==` and `===` usually behave the same. The surprising cases happen when types differ.

But even strict equality has one special case:

```js
NaN === NaN // false
NaN == NaN // false
Number.isNaN(NaN) // true
```

`NaN` is the only JavaScript value that is not equal to itself. That is why `Number.isNaN(...)` exists.

## `null` and comparisons

`null` has its own set of quirks:

```js
const x = null

x > 0 // false
x < 0 // false
x == 0 // false
x >= 0 // true
```

This looks broken until you remember that equality and relational comparisons do not use exactly the same coercion rules.

Another famous oddity:

```js
typeof null // "object"
```

This is not because `null` is secretly a real object. It is a historic bug that was never changed for compatibility reasons.

## Truthiness is not the same as loose equality

These two lines are both true:

```js
"0" == false // true
false == "0" // true
```

But this still runs:

```js
if ("0") {
  console.log("foo")
}
```

It prints `"foo"` because `"0"` is a non-empty string, and non-empty strings are truthy. Loose equality and boolean evaluation are different mechanisms.

## Other odd but explainable examples

```js
Math.max() < Math.min() // true
Math.max() > Math.min() // false
```

That happens because:

- `Math.max()` with no arguments returns `-Infinity`
- `Math.min()` with no arguments returns `Infinity`

And:

```js
typeof NaN // "number"
```

`NaN` means "not a number", but in JavaScript it still belongs to the numeric type.

## A few more classics

```js
1 < 2 < 3 // true
3 > 2 > 1 // false
```

Why?

```js
1 < 2 // true
true < 3 // 1 < 3 // true

3 > 2 // true
true > 1 // 1 > 1 // false
```

And this one:

```js
65 == [65] // true
```

The array is converted to the primitive `"65"`, then loose equality continues from there.

## `parseInt` and scientific notation

Another good example:

```js
parseInt(0.0000005) // 5
parseInt(0.00000005) // 5
```

This surprises people because `parseInt` first converts the number to a string. Very small numbers may be represented in scientific notation like `"5e-7"`, and `parseInt` stops parsing when it reaches `e`.

## Floating point is not a JavaScript-only problem

```js
0.1 + 0.2 === 0.3 // false
0.1 + 0.2 // 0.30000000000000004
```

This is not JavaScript being uniquely weird. It comes from IEEE 754 floating-point representation, which many languages use.

## Final thought

JavaScript can absolutely be surprising. But many of these behaviors are not random at all. They are the result of coercion rules, legacy decisions, parsing context, and compatibility constraints.

You do not need to build codebases full of expressions like `[] == ![]`. In real projects, weird behavior often appears because the code design is unclear, not because the language woke up in a bad mood.

Still, learning these edge cases is useful. It sharpens your understanding of the language and gives you a better mental model, which usually means fewer bugs and faster debugging.
